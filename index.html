What the system is

One CSV row = one OFP/CFP computation. Grouped per flight, it answers the questions a Flight Planning Systems Manager owns.

AOSUSER=lido · automated FIRST/ZFW/ACFT/FINAL triggers
→
HUMANTYPE=IFPL · qrXXXXX at offices D01–D13 / M01–M09
→
VALIDATEIFPUV · Eurocontrol ACK / REJ
→
FILELAST_FILED · ATS_HISTORY actions
→
BRIEFLAST_SENT to crew

It decides real things: is the AOS automation trustworthy, do dispatchers add value or burn fuel when they override it, what fraction of filed routes the LIDO engine actually authored (one lidoContributionData shared across 5 tabs), how many dispatchers are needed per hour, and which city pairs are payload-/weight-constrained and expensive. Nearly every table ranks, scores, or classifies.

High-level conclusions

The headline read at audit time: strong instrument, three structural risks (since remediated), one cross-system truth.

RISK 01

Silent wrong numbers

The monolith grew until later code overwrites earlier code: parseTimeToMinutes was redefined 4,600 lines apart with an incompatible format, so Route Analysis "Time Impact" read 0 for every flight. No test harness, no schema check, no assertion layer to catch it. Fixed in P1; harnesses now guard it.

RISK 02

Multi-day identity confusion

Core key is CALLSIGN_DATE_STD, but three analyses fell back to bare CALLSIGN — correct for one day, silently wrong once weekly files are merged (which the Workforce tab explicitly encourages). QR001 across 7 days collapsed into one flight. Fixed in P1 #7 (composite flightKey).

RISK 03

Environment coupling & trust

Chart.js / xlsx / Google-Fonts CDNs reintroduced online dependency the 93-col build avoided — Payload charts died offline on a corporate PC. Everything renders via innerHTML with unescaped log fields. Fixed in P3 — deps vendored inline, esc() at template boundaries.

STRENGTH

Where its analytical leverage is real

The center of gravity — the adopted / same-name / ≥90%-similar / truly-modified decomposition feeding one shared lidoContributionData across five tabs — is the right abstraction. The "fair comparison" ZFW-gating refuses to blame dispatchers for fuel deltas caused by load changes; the dispatcher-intent taxonomy, multi-file smart-merge with workflow fingerprints, the weekly workforce view, and the glossary-as-documentation are all above the bar for a single-file internal tool.

Dispatcher efficiency — the shipped staffing model

The headline workforce feature: "how many dispatchers does this operation need per shift" — computed from the log, not guessed.

PROCESSINGwall-clock UNION of [CALC_TIME, CALC_TIME+ELAPSED_TIME] per dispatcher
→
WINDOW12 h × active LOCAL-time roster blocks · Doha UTC+3
→
IDLEwindow − processing → per-shift staffing signal
  • Processing = interval union, not Σ ELAPSED. Each dispatcher's [CALC_TIME, CALC_TIME + ELAPSED_TIME] intervals are merged on the wall clock, so parallel SUFFIX/route calculations count once. A plain Σ ELAPSED over-counts by ~26.5% on the reference dataset.
  • Working window = local-time roster blocks. 12 h per active block, Doha local (UTC+3): day 06:00–18:00, night 18:00–06:00. The night shift is one block across midnight — never split per calendar day.
  • ATS actions folded in. ATS_HISTORY filing/send actions enter as point events, surfaced in a dedicated "ATS Actions" column alongside the calc workload.
  • Idle = window − processing, reported per dispatcher per shift.

Honest caveat: processing measures machine calc time only — briefing, coordination, phone calls and monitoring are invisible to the log. Idle% is therefore an optimizer-utilization indicator, not a measure of dispatcher busyness.

P1 · fuel $ · NEW

Missed & capped tankering

~$4M/yrillustrative upper bound

GAIN/LOSS (USD/ton) sit on every OFP whether tankering ran or not; SAVINGS only when executed. So flights that were favorable but not tankered (TANKERING_REASON=NO/blank) and MAXB/MAXH-capped ones are directly countable. Break-even documented (ZFWCF≈0.027/hr).

Compute queue × batch pruning

57 minpeak CALC_DELAY tail

Pure CPU queue wait is normally 25 s but reaches ~57 min under load — a late-OFP hazard near STD. Demand-side fix within your AOS-config authority: prune FREEOPT/OPTI candidates that never win per city pair + kill recurring error-burners (e.g. the RAPET–MASEK segment defect).

Frees the CPU headroom that makes P2 free
P2 · governance · extend

File-then-optimize gate

~1:1Changed per initial filing

AOS files a crude PRED route ~3 min after compute start, optimizes 20–58 min, then CHGs it — yet the evidence shows 9.75 h of lead time. 484 Filed+Sent vs 447 Changed/day. A one-line policy — hold filing until the batch completes when Hrs_Before_STD > threshold — cuts ATC churn and aligns filed-vs-optimized.

Serves the RAD / planned-vs-flown agenda · costs nothing
